# Hope Agent 0.13.0 — Scheduled Tasks Overhaul

> [简体中文](https://github.com/shiwenwen/hope-agent/blob/v0.13.0/docs/release-notes/v0.13.0.md) · **English**

> Released: 2026-06-27

> See [`CHANGELOG.md`](https://github.com/shiwenwen/hope-agent/blob/v0.13.0/CHANGELOG.md#0130---2026-06-27) for the full list.

The headline of v0.13.0 is a systematic overhaul of **scheduled tasks (cron)**: time zones finally take real effect with automatic DST handling; each task can now set its own permission and sandbox modes and can access your knowledge space; and a set of reliability knobs — a global concurrency cap, per-run timeouts, and a catch-up window for one-shot tasks — round it out. Conversations produced by scheduled tasks are lifted out of the main list into a new "History" view inside the task panel, with an unread badge in the sidebar. This release also adds unread indicators on the app icon (Dock) and for IM sessions, more useful yet more restrained desktop notifications, and a set of security tightenings around delivery and deletion.

## Highlights

- **Time zones finally take effect**: cron expressions (such as "every day at 9am") now fire in the time zone you choose and handle daylight saving automatically; new tasks default to the time zone detected from your browser, and the calendar preview matches actual firing. Previously the time-zone setting was ignored and everything fired in UTC (a UTC+8 user's "9am" actually fired at 5pm). After upgrading, existing scheduled tasks are corrected once to the local time zone, and affected firing times will shift accordingly.
- **Per-task permission and sandbox modes**: each task can now pick its own permission mode (follow Agent / standard / smart / YOLO) and sandbox mode (follow Agent / off / standard / isolated / workspace / trusted) instead of passively following the chosen Agent. Under unattended runs, "smart" mode lets a judge model allow operations consistent with the task's intent (including deletions or outbound sends the task itself asks for) while rejecting out-of-scope or seemingly injected actions; protected paths and dangerous commands are always blocked. To let a task run autonomously, prefer confining its blast radius in a sandbox over running YOLO bare on the host (that combination shows an explicit risk warning); a task configured with a non-"off" sandbox that finds Docker unavailable at runtime fails and records the reason — it never falls back to host execution.
- **Scheduled tasks can now access the knowledge space**: while running, scheduled tasks can use the knowledge-space note tools (`note_*`, `[[note]]` references, knowledge retrieval) and read the knowledge bases attached to that task's session or its project — the same access you have when chatting on the desktop yourself. Previously scheduled tasks were misclassified as IM-channel identities and uniformly denied, so note-dependent tasks silently received nothing; incognito sessions still have no knowledge-space access.
- **Scheduled-task conversations centralized in the task panel, with an unread badge**: conversations produced by scheduled tasks no longer mix into the main conversation list; they move into a new "History" view in the task panel — the left column aggregates every task's run history in reverse-chronological order, the right column shows the selected conversation read-only, with no need to open each task's detail page. When there are new results, the scheduled-task entry icon in the sidebar shows an unread badge, clearable via "Mark all read" at the top of the History view or by right-clicking the icon. Task detail, the task list, and the calendar also surface more information — delivery targets (stale targets in red), each run's delivery result, the executing Agent, per-run timeout, and consecutive-failure counts — with "Load more" pagination when there are many entries.
- **Unread indicators on the app icon and for IM sessions**: when there are unread messages, the app icon (macOS Dock) shows a total unread badge so you can tell at a glance even when the window is minimized or in the background; sessions from IM channels are marked in the sidebar with a distinctly colored unread badge to set them apart from regular desktop sessions, so you don't miss new IM-side messages when using desktop and IM at the same time.

## Scheduled-task reliability

- **Concurrency cap**: a new "max concurrency" setting (Settings → Scheduled Tasks, default 5, 0 = unlimited) avoids dozens of tasks firing at once and overwhelming the machine or tripping rate limits; over-the-cap due tasks keep their schedule and catch up on the next check, while manual "Run now" is unconstrained.
- **Per-run timeout + auto-disable notice**: a new "per-run timeout" setting (default 5 minutes, 30s–2h) abandons an over-running task and frees its concurrency slot; when a task is auto-disabled after consecutive failures, a desktop notification now fires with the reason (timeout / configuration error / repeated errors), and only genuine run failures count toward the disable threshold.
- **Catch-up window for one-shot tasks**: a new "catch-up window" setting (default 5 minutes, 0 = no catch-up) lets a one-shot task whose firing moment landed while the app was closed still run after restart, as long as it is no more than the window overdue; one-shot tasks that crashed mid-run are marked "missed" after restart (not re-run) instead of getting stuck "never firing."
- **More reliable delivery**: delivery to IM channels retries with backoff (up to 3 times) on failure, run logs gain a delivery-status marker, a delivery target whose channel account was deleted is flagged stale in red (with a heads-up before account deletion about which tasks reference it), and a new per-job "prefix delivery with task name" toggle helps tell apart multiple tasks delivering to the same chat.
- **More reliable cancellation, crash traces, and no faking success on empty output**: a cancel request is no longer swallowed in the instant a task is starting up, won't hit the next round of a recurring task by mistake, and won't discard an already-finished result as "cancelled"; a run that crashed or was force-killed is finalized as "error (interrupted)" and kept in run history after restart. A task that succeeds but produces no text is marked "no output" and skips delivery rather than sending an empty message; failure notifications carry a reason category, calendar matching is more precise for high-frequency tasks, and the dashboard success rate counts only "success vs. failure."
- **Invalid schedules rejected uniformly in the backend**: when creating or editing a scheduled task, invalid firing times, recurring intervals under one minute, and bad cron expressions or time zones are now rejected by the backend with a clear error — no longer storable as a "never fires" or "errors on every fire" bad task (previously only the model-tool path validated this, and the desktop / HTTP interfaces could bypass it).
- **Settings split into their own page + live in-progress feedback**: global scheduled-task settings move into a new "Scheduled Tasks" section on the settings page, with a gear shortcut at the top of the panel; while a task runs, the run history shows a live blue "running" entry that updates in place to success / failure / cancelled on completion.

## Notifications and experience

- **More useful yet more restrained completion notifications**: desktop chat-completion notifications now show a preview of the last assistant reply by default, with the session title as the notification title, so you can tell from the system notification which conversation finished; if you turn off reply-body previews in notification settings, both title and body switch to a fixed localized completion notice, keeping conversation content out of the system notification center.
- **Cleaner false triggers and fallback copy in notifications**: the hard-coded startup welcome notification is removed; session-pin and Agent-reorder save failures now surface as in-app toasts instead of system notifications.
- **Background-completed replies no longer silently marked read**: when you navigate away after sending a message, or a background tool / subagent completes and injects into a non-current session, that session correctly shows an unread indicator; sidebar group unread totals, project unread badges, and the global unread dot now use one consistent rule.
- **Menu-bar session list closer to the in-app sidebar**: the tray / menu-bar session area now uses a two-line summary showing status plus Agent, model, update time, and message count; with no in-progress sessions it shows the 5 most recent regular sessions. Replacing the native menu structure is deferred while the menu is open, so list updates no longer auto-close an already-open menu on macOS.
- **More restrained browser-extension guidance banner**: the Chrome-extension guidance banner on the chat page now animates smoothly in and out, and the delayed re-check after startup or a browser action is extended from 6s to 30s to reduce false prompts during the extension's reconnect window.

## Security

- **Safer scheduled-task delivery and deletion**: scheduled tasks can only deliver results to IM conversations you have already recorded (the model cannot conjure an unfamiliar chat to send to), and deleting a scheduled task requires your confirmation by default (smart mode leaves the call to the model, full-access mode skips it).
- **Tighter sandbox-mode approval boundary**: when `workspace` / `trusted` sandbox modes reduce approvals for `exec` edit commands, they no longer check only whether `cwd` is inside the workspace; they now also validate recognizable target paths in the command, `/workspace/...` container paths, redirection targets, and common file-command operands, so out-of-scope or dynamically expanded paths keep triggering approval.
